Overview

A smart contract on Ethereum decides who is an Owner, Moderator or Regular User. This API puts a conventional web layer in front of it, so any application can integrate with standard HTTP requests instead of talking to the blockchain directly.

Permissions are enforced on-chain, not by this API. The contract's onlyRole guards are the real gate — a caller without the right role is rejected by the blockchain itself, even if they bypass this service entirely.

Permission tiers

OWNER

Full control — register and remove users, promote moderators, edit system settings, plus everything below.

MODERATOR

Deactivate and reactivate users, approve content, read the user count — plus all user-level reads.

REGULAR_USER

Self-service actions. Handled in the dApp with the user's own wallet, not through this API.

NONE

No assigned role. Can still read all public data — roles, profiles, settings, status.

Architecture

A stateless service that translates HTTP into contract calls. There is no database — the contract's on-chain storage is the single source of truth.

Authentication

Reads are open. Writes and admin reads require an API key.

READS — no auth

Role lookups, profiles, settings and status call view functions. Free, instant, no key needed.

WRITES — API key

Send x-api-key. The server signs the transaction with its operator wallet and pays the gas.

HTTP header 
x-api-key: YOUR_API_KEY

Fail-closed by design. If an instance has no operator wallet configured it runs read-only and every write returns 503. It can never become an unauthenticated relayer for on-chain admin actions.

Quickstart

Read a role, then register a user. Pick your language.

# Read a wallet's role — open, no auth
curl HOSTX/api/v1/roles/0xd7C1D2D96cc9cD8D9e90876f77c1DBeC8A33B857

# Register a user — needs an API key; the server signs the transaction
curl -X POST HOSTX/api/v1/users \
  -H "content-type: application/json" \
  -H "x-api-key: $API_KEY" \
  -d '{"address":"0x7E2C9...85d7d","username":"alice"}'
const BASE = "HOSTX";

// read — no auth
const { role } = await fetch(`${BASE}/api/v1/roles/${address}`).then(r => r.json());

// write — API key required
const res = await fetch(`${BASE}/api/v1/users`, {
  method: "POST",
  headers: { "content-type": "application/json", "x-api-key": API_KEY },
  body: JSON.stringify({ address, username: "alice" }),
});

const { txHash, explorerUrl } = await res.json();
import requests

BASE = "HOSTX"

# read — no auth
role = requests.get(f"{BASE}/api/v1/roles/{address}").json()["role"]

# write — API key required
res = requests.post(
    f"{BASE}/api/v1/users",
    headers={"x-api-key": API_KEY},
    json={"address": address, "username": "alice"},
)
print(res.json()["explorerUrl"])

Errors & limits

Every failure returns the same JSON envelope with a machine-readable code. On-chain reverts are translated into proper HTTP statuses — raw hex is never returned.

Error envelope 
{
  "error": {
    "code": "bad_request",
    "message": "'address' must be a valid Ethereum address.",
    "details": { "field": "address" }
  }
}
StatusCodeWhen
400bad_requestInvalid address, missing field, malformed JSON
401unauthorizedMissing or wrong API key
404not_foundUnknown route, or the address isn't registered
409conflictContract reverted — e.g. “User does not exist”
429rate_limitedOver 120 requests per minute per IP
502upstream_errorBlockchain RPC unreachable
503service_unavailableWrites disabled, or operator out of gas

Rate limit: 120 requests per minute per IP. Writes wait for one block confirmation (~12s on Sepolia) before responding, so set a client timeout of at least 30 seconds.
